# Extension authentication threat model

Runtime extensions are trusted executable Node.js code. Installing one grants it the normal filesystem, process, environment, and network authority of the Rigyn process. The package trust prompt, immutable package records, and project-package lock are therefore the primary defense against malicious extension code; the authentication broker is not a JavaScript sandbox.

The broker protects provider credentials from accidental exposure through the ordinary extension API. API keys, OAuth tokens, cloud secret keys, signed headers, and credential-store records are not returned by brokered fetch, ordinary lifecycle events, RPC, session state, diagnostics, errors, or tool results.

An extension may register public authentication metadata and an exact request-origin allowlist. `api.auth.fetch` then enforces all of the following before attaching authority:

- the caller owns the current provider registration generation;
- the URL uses an allowed exact origin;
- caller-supplied authentication headers and redirects are rejected;
- the registered API-key, bearer, or AWS SigV4 strategy matches the resolved credential;
- request and response bodies are bounded and cancellation remains observable;
- refresh, logout, profile changes, and reload are resolved by the host for each request.

Authentication is intentionally provider-scoped rather than session-scoped. Session IDs and profile selectors are not accepted by `api.auth.fetch`, and extension requests cannot retarget the active credential by adding request metadata. Each call re-resolves the host-selected credential for the registered provider; profile selection remains an owner-level provider operation shared by that provider's sessions.

Rigyn also has explicit high-trust capabilities for extensions that genuinely need to implement provider or host integration. They are disabled unless the package is trusted and its manifest declares the individual permission:

- `nativeUi` enables decoded input interception, editor and autocomplete replacement/wrapping, persistent structural mounts, paste, and resolved theme access. It does not expose terminal byte streams or permit unvalidated ANSI output.
- `unsafeTerminal` enables raw terminal input interception and arbitrary terminal-protocol writes. It can observe secret input, consume host shortcuts, corrupt display state, and emit terminal control protocols. Grant it only to fully reviewed local or managed packages; disposal and generation replacement revoke the handle but cannot undo bytes already emitted.
- `credentialAccess` enables `api.native.credentials.resolve` and registration of `managed_oauth` provider methods. The native snapshot returns the active API key, bearer/OAuth access token, ambient-identity description, and derived request headers without an OAuth refresh token. Managed login/refresh callbacks are a higher-authority contract and receive the OAuth values required to own that provider protocol, including its refresh token. Neither surface receives a credential-store handle.
- `providerWire` enables `api.native.providers.intercept`. It can transform JSON request bodies, secure request destinations, and request headers before transmission and can inspect complete HTTP response headers. Credential-bearing request headers and URL parameters are removed from the observation presented to the interceptor; a host-applied `baseUrl` patch preserves private query values without revealing them, while an interceptor can still intentionally replace a complete URL or outgoing header. Response bodies are never exposed. WebSocket frame contents are transformable only when they are outbound JSON; inbound frames expose bounded metadata rather than content.
- `providerOverride` enables generation-owned complete replacement or field-level overlay of an existing provider ID. An overlay may change presentation, destination, headers, catalog, or streaming implementation while preserving unspecified authentication and behavior. Disposing or unloading the owner restores the exact remaining composition.
- `sessionRaw` enables bounded canonical event/run pages, optional provider-neutral compacted context, and process-local unredacted prompt snapshots for an explicitly selected session branch. It does not expose a mutable store handle or widen workspace ownership.
- `hostConfiguration` enables effective host path/settings snapshots and validated user or trusted-project configuration updates. It is not an arbitrary file-write capability.

These capabilities are convenience and lifecycle contracts for code that already has the process authority to read files, environment variables, and the network. A manifest permission does not sandbox that code or make an untrusted project executable. It makes exceptional authority reviewable, individually gated, ownership-scoped, and automatically reversible on failed activation, reload, or disposal. Native values and handles become stale when their generation aborts. Credential snapshots and managed-auth callback values must remain ephemeral and must never enter logs, diagnostics, session state, extension output, or external requests other than the reviewed integration that required them. Managed callback results are normalized and bounded, projected secrets are registered with redaction, and their registrations are removed with the owning activation generation; those safeguards reduce accidental leakage but do not make provider secrets safe to persist.

This boundary limits accidental leaks and confused-deputy requests through the supported API. It cannot contain a malicious installed extension that reads host files or environment variables or opens its own network connection. Untrusted extensions require an external execution backend or operating-system isolation chosen by the owner.

Focused verification lives in `test/auth/authenticated-fetch.test.ts`, `test/extensions/runtime.test.ts`, `test/extensions/runtime-api-semantic-conformance.test.ts`, `test/extensions/runtime-native-security.test.ts`, `test/providers/wire.test.ts`, and the provider/reload tests. It covers exact-origin enforcement, permission and trust denial, ownership and stale-generation rejection, failed-activation rollback, caller and generation cancellation, reload/unload cleanup, caller header rejection, body bounds, refresh/logout behavior, pre-sign provider transformations, WebSocket limits, and secret-free ordinary results.
