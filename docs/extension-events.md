# Runtime extension events

This is the complete reference for names accepted by `api.on(name, listener)`. The public TypeScript contracts are `RuntimeExtensionEvent`, `RuntimeExtensionEventMap`, and `RuntimeExtensionEventResultMap` from `rigyn/extensions`.

## Listener contract

Listeners are committed only after their runtime entry activates successfully. For a given event, they run one at a time in extension load order and then registration order. Each listener receives a bounded clone of the current payload and a `RuntimeExtensionListenerContext` containing the owning extension identity, workspace, generation-scoped `signal`, and UI surface. When the event carries host-owned session identity, the context also contains callback-bound `session` identity and safe actions for inspecting the session, model, usage, and redacted system prompt; checking idle/queued state; aborting; or compacting. Those actions are generation- and callback-signal-bound and never expose a raw store handle. The pre-trust event instead receives `RuntimeProjectTrustListenerContext`, whose UI contains only `hasUI` and `confirm`. A listener cannot change another listener's payload by mutating its argument.

The host accepts at most 128 `tool_call` listeners in aggregate because every such listener may transform input and every effective transform requires one coordinator audit entry. A 129th registration fails transactionally; listeners for other event types are unaffected.

The tables use these failure policies:

- **Observer:** the return value is ignored. Dispatch attempts every registered listener, records a redacted diagnostic for each failure, and rejects after the last listener when called directly. Harness service observers catch that rejection because an observation cannot roll back durable work. TUI and RPC hosts may also surface a warning. Cancellation aborts the listener context and ends the dispatch; it does not undo an already committed event.
- **Transform:** listeners reduce in order. Each valid result becomes the next listener's input. A non-cancellation exception or invalid result is diagnosed, leaves the accumulated value unchanged, and allows later listeners to run. Cancellation propagates to the operation that requested the transform.
- **Session guard:** the first result with `cancel: true` stops the chain. `reason` is user-visible context, not a separate cancellation switch. A non-cancellation failure is diagnosed and otherwise fails open; cancellation of the supplied signal aborts the host operation.
- **Tool guard:** the first result with `block: true` stops the chain. A listener failure fails closed and produces a blocked tool result. Cancellation propagates through tool coordination.
- **Collector:** valid paths are appended in listener order. A failed listener contributes nothing and later listeners continue. Caller or generation cancellation aborts the whole discovery pass.
- **Trust decision:** user-scope and explicitly requested invocation extensions run in order before protected project resources are read. The first `yes` or `no` result wins; `undecided`, invalid results, and listener failures continue to the next listener. `remember: true` persists only the exact canonical workspace.

All structural results are validated and bounded by the host. Do not put credentials in event payloads, return values, errors, UI messages, or durable state.

## Deterministic lifecycle order

Before project trust is known, user-scope and explicitly requested invocation entries activate and may receive `project_trust`. CLI overrides and workspaces without protected resources suppress this event. An extension decision precedes saved policy, the configured default, and the built-in chooser. The same active host is handed to runtime construction; after approval, project entries append without activating the earlier entries again. After all candidate entries have been attempted, `resources_discover` runs before skills, prompts, and themes are finalized. The interactive or RPC host then emits `session_start` when it opens a session.

A successful reload builds and discovers the candidate generation first. The active generation then receives `session_end` with reason `reload`; the swap commits; the old host receives `session_shutdown` and disposes its registrations in reverse order; and the candidate receives `session_start` with reason `reload`. If the swap fails after the old session ended, the old generation receives `session_start` with reason `reload_rollback`, while the rejected candidate is shut down.

For a normal agent run, the stable high-level order is:

```text
model_select
thinking_level_select
event(run_started)
agent_start
before_agent_start

for each provider step:
  [session_before_compact -> event(compaction_started)
   -> event(compaction_completed) -> session_compact]
  context
  before_provider_request
  turn_start
  message_start
  message_update *
  after_provider_response
  message_end(assistant)
  [tool_call -> tool_execution_start -> tool_execution_update *
   -> tool_result -> tool_execution_end -> message_end(tool)] *
  turn_end

event(run_completed | run_failed | run_cancelled)
agent_end
agent_settled
```

`event` is emitted for each append-only durable envelope after any typed projection derived from that same envelope. Therefore, for example, `message_update` precedes `event(text_delta)`, and `tool_execution_end` precedes `event(tool_completed)`. `message_end` is a transform applied to every canonical user, injected, assistant, and tool message before persistence; it is not paired only with the streamed assistant `message_start`. For the provider-finalized assistant message it additionally carries bounded `finalized` response fields. `message_start` and every `message_update` include the accumulated provider-neutral assistant stream. A finalized tool call produces `message_update` with `kind: "tool_call_end"` after its canonical assistant message is committed. On a successful provider step, `turn_end` is held until the assistant completion is durable and, when the step requests tools, until its final model-visible tool-result message is durable. A failed step emits `turn_end` with a failed outcome. Terminal cleanup flushes any remaining turn before `agent_end`, and `agent_settled` always follows `agent_end` for completed, failed, and cancelled runs.

Every run-scoped typed payload implements `RuntimeRunScope`: immutable `threadId`, `runId`, and the exact resolved `branch`, plus `step` when the boundary belongs to a provider step. The listener's top-level event object is frozen. Nested structural data remains governed by the event-specific transform rules below; extensions must use the host-owned scope instead of a mutable active-session pointer. Extension child runs started inside a run-scoped listener, generic durable-event listener, model-selection listener, session boundary carrying an owning or prospective thread, or child callback are attributed to that boundary's host-owned thread, so targeting another thread cannot bypass the no-nesting policy. Thread-like fields in model-controlled input and shared-event payloads never establish requester identity.

Host input, session navigation, shell commands, and explicit compaction occur outside the run loop and therefore enter at their named boundary rather than at a fixed position in that sequence.

## Choosing the right boundary

Use a guard only to prevent an unsafe host action. Use a transform when the package owns a bounded structural change. Use an observer for telemetry, UI, indexing, or cleanup that must not alter the action. Prefer `event` when rebuilding state from durable history, and prefer the typed event when the package needs a live boundary or normalized payload that is not itself durable.

Shared extension topics registered through `api.events.on()` are a separate bounded JSON message bus. They are generation-owned and deterministic, but they are not members of `RuntimeExtensionEvent` and are not lifecycle events.
